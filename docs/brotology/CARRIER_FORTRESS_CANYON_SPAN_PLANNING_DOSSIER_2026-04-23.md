## Current Determinations

The following points are now treated as current working truth for planning.

### 1. Site Read

`Grand Canyon` is not the literal target site.

It is the reference geometry carrying:

- long-span intuition
- terraced switchback relay logic
- inertial depth
- upper span with deep lower service volume

The first simulation grammar is therefore:

- `open_pit_mine / canyon_scale_basin`

The later translation target is:

- `urban_excavation_block`

### 2. Mechanical Read

The core mechanism is not a one-time domino effect.

It is:

- a `nested rotary cascade`
- over a suspended span
- with preloaded heavy members
- released by small timing triggers
- caught and redirected through bounded recharge paths

The key correction is:

- `micro_sequencer` triggers
- `charged_member` actually throws the motion

### 3. Architectural Class

The larger class is:

- `carrier_fortress`

Meaning:

- a mainframe fortress
- an omni-directional aircraft carrier for kinetic utility systems
- a launch / catch / recharge / recovery platform

The canyon-span machine is one deployment armature of this larger class, not the whole ontology.

### 4. Utility Read

The hot core or heat source is only justified if it is tied to real utility sinks.

Approved sink families:

- habitation heat and hot water
- drying and thermal conditioning
- water lifting or pumping assist
- milling, grinding, crushing, or process routing
- sterilization, cleaning, and waste handling

Rejected read:

- pure spectacle with no utility plant beneath it

### 5. Runtime Read

Champion Council can support a first simulation facility without inventing a second disconnected runtime.

Grounding:

- `static/main.js` already carries live 3D scene/workbench and world-export surfaces
- `server.py` already exposes command/help surfaces for structured control
- the repo already prefers facility-of-facilities composition over fake god-objects
- physics is an inhaled facility, not a baseline dependency
