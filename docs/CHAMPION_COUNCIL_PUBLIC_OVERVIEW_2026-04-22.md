# Champion Council Public Overview 2026-04-22

Champion Council is a local-first AI operator runtime.

It combines model orchestration, live observation surfaces, query-work, continuity recovery, planning docs, diagnostics, workflows, and MCP tools inside one system. The project is built for people who want more than a chat box and more than a tool API: it is the environment where AI work is run, inspected, steered, and recovered.

## What It Is

Champion Council is best described as:

- a multi-surface AI control room
- a local-first operator cockpit
- a runtime for live observation, planning, diagnostics, and recovery

It is not just:

- a model switcher
- a Hugging Face Space shell
- a workflow editor
- a docs viewer

## What Makes It Different

Most AI systems split execution, observation, planning, and recovery across separate tools. Champion Council keeps them in one runtime.

That means the same environment can carry:

- model councils and provider routing
- web and text theater views of runtime state
- embodiment and workbench staging
- blackboard-style query-work
- continuity re-entry after interruption
- queryable docs and memory
- workflow automation and MCP exposure

## The Main Surfaces

Champion Council currently includes:

- a council runtime for multi-model execution and comparison
- a web theater for spatial observation and interaction
- a text theater for compact consult, diagnostics, and blackboard reads
- a workbench for skeleton, scaffold, pose, and sequence staging
- a blackboard query lane for visible ordered work
- continuity and planning packets for re-entry and operator posture

## The Design Rule

Champion Council is organized around one rule:

- one truth source, no duplicate authority planes

Live runtime state stays primary. The theater, blackboard, continuity, reports, and docs help the operator see and carry that truth, but they do not replace it.

## Named System Lanes

Some of the most important named lanes in the current system are:

- `output_state` for readable live posture
- `Tinkerbell` for attention and pointer cues
- `Pan` for support and contact measurement
- `Dreamer` for bounded proposing and scoring
- HOLD for intervention and audit boundaries
- `docs_packet` and `continuity_packet` for planning and re-entry

These are not just labels in prose. They are part of the actual current system shape.

## What It Can Honestly Be Presented As

Today, Champion Council can be shown as:

- an operator runtime for AI systems
- a continuity-aware diagnostics and planning environment
- a multi-surface workspace for live AI observation and control
- a foundation for embodied, theatrical, and workflow-driven AI behavior

## Public Exposure Rule

Public presentation does not require public access to every operator lane.

For a public or `protected` Space posture, the public surface should be limited to:

- the visible app behavior
- selected public doctrine
- bounded interaction surfaces meant for demonstration

It should exclude:

- continuity restore lanes
- checkpoint and persistence controls
- FelixBag/file browsing and editing
- export/clone-grade routes
- operator-only debug, cache, and restart controls

## What It Is Not Claiming

Champion Council should not be described as:

- a finished AGI operating system
- a universal autonomous planner
- a single magic model wrapper

The stronger and more accurate description is that it is a real operator environment with live runtime, theater, continuity, docs, help, workflow, and diagnostics lanes already working together.
