## One-Sentence Description

Champion Council is a local-first AI operator runtime that combines model orchestration, spatial and text theaters, embodiment/workbench tools, blackboard query-work, continuity recovery, docs and memory planning, diagnostics, workflows, and MCP exposure inside one coordinated environment.

## Short Description

Champion Council is not just a model switcher, not just a workflow editor, and not just a Hugging Face Space wrapper. It is the active runtime used to observe, steer, diagnose, and recover a multi-surface AI system. The project combines live runtime state, operator-facing theater surfaces, bounded orienting state, continuity re-entry, queryable docs and memory, and automation surfaces into one local-first control room.

The accurate product read is: an AI operator cockpit with continuity, planning, diagnostics, and embodiment built into the same runtime.

## What It Actually Is Today

Champion Council currently exists as all of the following at once:

- a council runtime with up to 32 slots for plugged models and provider routing
- a web theater for spatial observation, staged embodiment, workbench interaction, and selected text-rendered overlays
- a text theater for compact render, consult, blackboard, evidence, and snapshot-facing reads
- a workbench/embodiment surface for skeleton, scaffold, attachment, pose, and authored sequence work
- a downstream skin/Coquina embodiment lane that projects shared truth into web materials, text-braille skins, hair fields, and trails
- a blackboard/query lane for visible ordered operator reasoning work
- an `output_state` carrier that makes live posture readable without becoming a second authority plane
- a continuity restore lane for reacclimation after resets and context compression
- a docs and FelixBag planning surface that can be queried and checkpointed
- a workflow and MCP exposure layer for automation and external tool use

## The Core Rule

Champion Council is organized around one architectural rule:

- one truth source, no duplicate authority planes

In practice, that means:

1. live runtime state is the authority
2. live theater and capture corroboration come next
3. blackboard/query-work and `output_state` are readable carriers, not replacement brains
4. reports and packets help the operator re-enter and plan
5. docs and archive continuity assist, but do not outrank live corroboration

This rule is what keeps the system coherent instead of collapsing into multiple competing planner surfaces.

## Named Runtime Systems

### Tinkerbell

Tinkerbell is the first-pass pointer and attention layer.

Its correct role is:

- point at the hot seam
- carry attention target and why-now pressure
- remain derived from the orienting field

Its incorrect role would be:

- replacing runtime truth
- mutating embodiment truth directly

### Pan

Pan is the support/contact/route measurement lane.

Its correct role is:

- measure support and contact truth
- stay grounded in physical or procedural posture
- later assist routing or proposal without bypassing truth

### Dreamer

Dreamer is the bounded proposer and scorer.

Its correct role is:

- consume truthful observation and orienting context
- propose candidate actions or interpretations
- score them without becoming a detached authority plane

The current code-backed reality is:

- Dreamer already reads meaningful orienting context
- Dreamer already consumes query-thread and oracle-style context
- Dreamer does not yet directly weight proposals from the Tinkerbell pointer itself

### HOLD

HOLD is the intervention latch and audit seam.

Its correct role is:

- mark human-intervention or stop points
- provide a visible and auditable intervention boundary

Current reality:

- HOLD is live as a tool and UI seam
- HOLD is not yet a first-class carried blackboard/query/output-state row family
