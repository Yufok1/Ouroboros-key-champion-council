## From Primitives to Places

**Quick Wins**

**Deliverable: glTF “asset-ref” pipeline for scene objects.** *What exists now:* the current entity["organization","Three.js","javascript 3d library"] theater renders geometric primitives and already persists/normalizes environment objects (kind/id/key, spatial fields, appearance, etc.) in the habitat substrate, while packaging/export is already live-verified per your bundle contract. *What needs to be built:* add an `appearance.asset_ref` (or `appearance.model_url`) field to the canonical object record and implement a client-side loader/cache that resolves `asset_ref → GLB/glTF` and instantiates the model under the object’s transform; include “material policy” (unlit vs PBR), “scale normalization,” and a deterministic “pivot rule” (e.g., ground-contact at y=0). *Implementation sequence:* (1) extend object schema to carry `asset_ref` and `collision`/`bounds` metadata; (2) integrate glTF loading and caching; (3) add model-instance lifecycle (spawn/replace/dispose); (4) update export/import so bundles carry referenced assets (or a manifest of CDN paths). glTF 2.0 is specifically designed for efficient delivery/loading of 3D scenes and assets (including meshes, materials, animations, cameras, lights). citeturn0search6 *User-facing result:* an operator can switch any object from “box/sphere” to a real prop/room module by selecting an asset in the Environment UI; the 3D view becomes “place-like” with recognizable architecture and props instead of debug primitives. *Risk/complexity:* straightforward but detail-heavy; biggest risks are asset hygiene (scale, pivots), texture formats, and keeping imports deterministic for packaging.

**Deliverable: minimum viable “room kit” using modular instancing + CC0 textures.** *What exists now:* stable camera, fog, ground plane/grid, labels, and a live sync bridge. *What needs to be built:* a tiny modular building system that can assemble a room from repeatable wall/floor/door segments (one wall segment mesh instanced N times) plus two or three CC0 PBR materials for stone/wood/sand. Instanced rendering is the primary quick win for “5–50 objects” because a single geometry/material can be drawn many times with different transforms, cutting draw calls. citeturn1search0 For ready-to-use CC0 ground materials, entity["organization","Poly Haven","cc0 3d asset site"] provides high-resolution stone floor textures under CC0. citeturn9search0 *Implementation sequence:* (1) define a `room_module` object kind variant (or `prop` subtype) that expands into 4 walls + floor; (2) implement InstancedMesh walls/floor; (3) add 1 directional light + ambient/hemisphere, enable shadow maps only for the room (optional); (4) add a “portal/door” module that is visually distinct. *User-facing result:* you can spawn “Room A” that feels like an enclosed space (walls, floor, doorway, lighting) while still being deterministic and easy to persist. *Risk/complexity:* low–medium; the main risk is visual quality vs performance (shadows, texture resolution).

**Medium Builds**

**Deliverable: fast picking/collision substrate using BVH and optional CSG door cutting.** *What exists now:* object selection and interaction are present conceptually (labels, kinds, actions). *What needs to be built:* accelerate raycasting and spatial queries using a mesh BVH so selection, “click-to-focus,” and collision checks remain responsive as you add higher-poly props and modular rooms. `three-mesh-bvh` provides BVH raycasting and other spatial queries for three.js meshes and is designed to dramatically speed raycasting. citeturn0search3 For “cut a doorway/window in a wall module,” `three-bvh-csg` provides boolean CSG operations on top of the BVH. citeturn0search1 *Implementation sequence:* (1) extend mesh build step to compute BVH once per static mesh; (2) override raycast for BVH-accelerated picking; (3) define collider policies per kind (`zone` uses bounds; `prop` uses BVH); (4) optionally add CSG as an editor-only tool to bake doorway variants. *User-facing result:* clicking objects stays instant, and “room modules” can have real door openings without hand-modeling every cut. *Risk/complexity:* medium; BVH integration is usually clean, but CSG robustness depends on manifold geometry and will need guardrails. citeturn0search1turn0search3

**Deliverable: explicit LOD + geometry merge pipeline for stable performance.** *What exists now:* object counts are still small, but the roadmap targets richer scenes. *What needs to be built:* (a) per-asset LOD groups (hi/med/low meshes) and (b) merge/static-bake for “distant clutter.” `THREE.LOD` is a built-in distance-based LOD switcher. citeturn0search4 For merging static geometry (when you decide multiple props become “one baked cluster”), `BufferGeometryUtils.mergeGeometries()` and `mergeVertices()` exist as supported utilities. citeturn14search0 *Implementation sequence:* (1) add `lod_levels` to `appearance` for assets that provide multiple meshes; (2) implement LOD instantiation (swap by camera distance); (3) add a “bake cluster” editor command to merge N static props into one mesh for far-field. *User-facing result:* the operator can mark objects as “LOD-managed” and see stable FPS as worlds become denser. *Risk/complexity:* medium; the correctness risk is mostly in bounding volumes and ensuring baked meshes remain selectable (you may need “selection proxies”).

**Research Problems**

**Deliverable: terrain generation that is editable, packageable, and doesn’t become a perf trap.** *What exists now:* a flat ground plane + grid. *What needs to be built:* either (a) heightmap-based terrain chunks or (b) procedural terrain generation with export to a deterministic artifact that can be bundled. `THREE.Terrain` is a dedicated procedural terrain library (with multiple generators and mesh scattering utilities). citeturn14search2 If you want determinism with seeded noise, `simplex-noise` is a common MIT-licensed approach for generating 2D/3D noise fields with optional seeded PRNG. citeturn14search1 *Implementation sequence:* (1) decide terrain representation (heightmap mesh vs procedural baked mesh); (2) generate + bake to a mesh artifact; (3) add collision proxies and simplified navmesh export. *User-facing result:* an operator can create “yard terrain” that is not just a plane and have it survive export/import. *Risk/complexity:* genuinely tricky once you require (i) editability, (ii) navmesh compatibility, and (iii) packaging determinism.

## Agent Sprites and Visual Identity

**Quick Wins**

**Deliverable: modality-specific billboard sprites integrated with existing CSS2D labels.** *What exists now:* council slots and object kinds already render as labeled nodes. *What needs to be built:* a sprite atlas (PNG/WebP) for modality icons (ears/voice/eyes/brain/compass) and a renderer rule: `slot.kind=agent` gets a `THREE.Sprite` plus the existing CSS2D label. A `Sprite` is explicitly designed to always face the camera. citeturn2search2 Three’s billboard pattern is well-supported and works cleanly with camera-facing badges/labels. citeturn2search4 *Implementation sequence:* (1) add `appearance.sprite_id` and `appearance.tint`; (2) load atlas once; (3) create sprites per agent; (4) keep CSS2D label anchored to sprite with a stable offset. *User-facing result:* the operator can tell “ASR vs TTS vs VLM vs LLM vs embedder” immediately without reading text. *Risk/complexity:* low; biggest risk is visual clutter—solve with scale rules and “zoom-level label collapse.”

**Deliverable: visual state machine for agents (idle/running/failed/learning).** *What exists now:* workflows and tool calls generate execution state; Dreamer training cycles exist in telemetry. *What needs to be built:* a small “agent render state” mapper: state → color/animation (pulse when running, red blink on error, slow glow when learning). *Implementation sequence:* (1) define canonical agent states; (2) add a per-frame animator that applies scale/opacity/particle emission; (3) mirror state to label (small status chip). *User-facing result:* you can watch the council “light up” as work happens, and spot failures instantly. *Risk/complexity:* low.

**Medium Builds**

**Deliverable: low-poly animated glTF agents with standardized animation hooks.** *What exists now:* 3D scene supports meshes and labels; you already have object persistence and actions. *What needs to be built:* a standardized “agent rig contract” (idle/walk/work/error) so any glTF character can be dropped in as an agent body, plus a loader/retarget path. `SkeletonUtils.clone()` exists specifically to correctly clone skinned meshes/bones when instancing characters. citeturn1search2 For CC0 starter characters, Kenney provides rigged/animated character packs under CC0. citeturn9search22 *Implementation sequence:* (1) adopt a minimal animation naming convention; (2) load glTF agent mesh once, clone per instance; (3) bind animation state to workflow/Dreamer status; (4) add a simplified “agent capsule collider” for navigation. *User-facing result:* agents become “characters” rather than icons, while still light enough for browser runtime. *Risk/complexity:* medium; animation retargeting and asset variability are the main pain points.

**Deliverable: VRM-based avatar option for richer identities.** *What exists now:* you have a council-based multi-model system that benefits from strong persona/identity cues. *What needs to be built:* an optional VRM loader path using `@pixiv/three-vrm`, which adds robust VRM support on top of the glTF loader. citeturn2search3 If you want a “Ready Player Me” ingestion path, their avatars are distributed as GLB files at unique URLs, with licensing terms that differ between non-commercial and registered commercial use. citeturn2search13turn2search12 *Implementation sequence:* (1) implement VRM loader plug-in; (2) map “talking” state to mouth/face animation (even if simplified); (3) expose per-agent avatar selection in the UI. *User-facing result:* “agents have faces,” which materially improves interpretability in demos and recordings. *Risk/complexity:* medium; VRM is mature, but you’ll need strict asset-size limits and fallback rendering modes.

**Research Problems**

**Deliverable: modality-as-VFX (sound fields, gaze cones, thought clouds) that remains readable at scale.** *What exists now:* ambient particles and theme skins exist. *What needs to be built:* a coherent VFX grammar that (a) communicates modality and (b) communicates state, without turning the scene into noise. *Implementation sequence:* design iterations, user testing, and performance tuning. *User-facing result:* you can “read” the room like an instrument panel. *Risk/complexity:* medium–high; it’s UX research as much as engineering.

## Agent Simulation and Autonomy in 3D Space

**Quick Wins**

**Deliverable: workstation anchoring + “walk-to-work” illusion without navmesh.** *What exists now:* facilities/workflows exist and environment objects include kinds like actor/node/workflow/portal/zone. *What needs to be built:* represent each facility as a “workstation” object with an anchor transform; when a workflow node runs, assign it to an agent slot and move that agent toward the workstation along a straight-line or spline path, with simple obstacle avoidance rules. *Implementation sequence:* (1) add `bindings.facility_id` to workstation objects; (2) add a movement component to agents; (3) on workflow node start, set target workstation; (4) on completion, spawn/update an artifact object at a designated “output shelf.” *User-facing result:* workflows stop feeling abstract—operators see “who is doing work where.” *Risk/complexity:* low; it’s mainly glue between workflow events and scene animation.

**Deliverable: steering behaviors for “wander / approach / avoid” with deterministic fallback.** *What exists now:* you can represent agents as entities in the scene. *What needs to be built:* either implement basic steering yourself or use entity["organization","Yuka","javascript game ai library"], which provides autonomous agent design, steering behaviors, and navigation mesh support as an engine-agnostic AI layer. citeturn2search0turn2search16 *Implementation sequence:* (1) add a Yuka “vehicle” per agent; (2) map environment objects to obstacles/targets; (3) keep a deterministic “no-steering” fallback so packaged worlds remain stable. *User-facing result:* agents can roam, gather, and “line up” at facilities in a believable way. *Risk/complexity:* low–medium; the biggest risk is making behavior look intentional rather than random.

**Medium Builds**

**Deliverable: navmesh pathfinding for real locomotion.** *What exists now:* rooms/props will become obstacles as you move beyond primitives. *What needs to be built:* an authored navmesh (Blender → export GLB) plus a runtime path planner. `three-pathfinding` supports pathfinding across navmeshes, zones, and step clamping, and is designed for three.js geometry inputs. citeturn1search1 *Implementation sequence:* (1) decide on navmesh authoring workflow; (2) load navmesh as hidden mesh; (3) compute path A→B and move agent along it; (4) add “off-mesh links” later for doors/teleports. *User-facing result:* agents move around walls and through doors, enabling credible facility layouts. *Risk/complexity:* medium; navmesh maintenance becomes part of world authoring.

**Deliverable: behavior runtime that maps workflow DAGs to spatial behaviors.** *What exists now:* you have a workflow engine (DAG executor) that can run tool/agent/http/etc nodes. *What needs to be built:* a behavior orchestrator that converts “workflow node executing” into “agent is at workstation performing action,” plus interrupt/replan hooks. `behavior3js` provides MIT-licensed behavior tree primitives and supports controlling many agents from shared trees. citeturn12search0 For deterministic UI-driven orchestration (especially for operator intervention), a state-machine approach like XState offers actor/statechart patterns and strong tooling. citeturn12search1 *Implementation sequence:* (1) define an “agent task API” (MoveTo, ExecuteTool, WaitForResult, Report); (2) implement leaf tasks; (3) map workflow node types to leaf tasks; (4) expose operator interrupts (pause, reroute, force-complete). *User-facing result:* the operator can watch an execution “play out” spatially and intervene without breaking provenance. *Risk/complexity:* medium; the hard part is interrupt semantics and keeping provenance graphs coherent.

**Research Problems**

**Deliverable: replanning autonomy (utility/GOAP-like facility routing) that stays legible.** *What exists now:* Dreamer learns from usage and workflows can branch/if/merge. *What needs to be built:* a “world-state” representation that allows agents to choose among facilities dynamically (e.g., if armory busy, go to training; if injured, go to medicus). *Implementation sequence:* experiment with utility scoring, planning horizons, and operator overrides; ensure decisions are auditable. *User-facing result:* a living facility ecosystem rather than scripted movement. *Risk/complexity:* high; autonomy that feels smart (and remains explainable) is one of the core hard problems.

## The Dreamer Learning Loop in World Context

**Quick Wins**

**Deliverable: define a “spatial observation vector” and start logging it with rewards.** *What exists now:* Dreamer already learns from tool-call rewards; environment interactions exist as structured events; you have a concept of spatial state (object x/y/scale/tilt) and execution traces. *What needs to be built:* an explicit observation encoder that converts environment state into a fixed schema: e.g., `(agent_pos, target_facility_pos, distance, current_node_type, object_density, last_action_type, success/failure)` and store it alongside reward. Dreamer-family agents learn policies inside a learned world model and actor-critic trained on imagined trajectories. citeturn6search6turn7search1 *Implementation sequence:* (1) define observation schema v1; (2) log it for every env_control and workflow step; (3) retrain Dreamer on this augmented dataset; (4) evaluate whether it predicts outcomes like “time-to-complete” or “failure probability.” *User-facing result:* “Dreamer understands the room” begins as predictive metrics (not yet autonomous design). *Risk/complexity:* low–medium; measurement design is the key.

**Deliverable: add layout-efficiency reward shaping without changing core task semantics.** *What exists now:* rewards exist, but may not encode spatial efficiency. *What needs to be built:* secondary reward terms for travel distance/time, number of tool invocations, and reroutes. DreamerV3’s published framing emphasizes robust learning across diverse tasks using world models and imagination, suggesting you can profitably learn from structured signals if they’re stable. citeturn7search1turn7search12 *Implementation sequence:* (1) define efficiency metrics; (2) log as auxiliary rewards; (3) monitor whether Dreamer policy/value predictions correlate with better layouts. *User-facing result:* you can quantify “this facility layout is better” and surface it in the UI. *Risk/complexity:* low; the risk is reward hacking if metrics are poorly designed.

**Medium Builds**

**Deliverable: RL-guided layout optimization as a formal MDP (“designer agent”).** *What exists now:* you already have object placement, workflows, and measurable outcomes. *What needs to be built:* a training environment where actions are “place/move/remove object,” and episodes evaluate operational throughput (workflow completion, errors, latency). PCGRL explicitly frames level design as a sequential decision process (MDP) and trains a generator via RL to maximize final content quality under constraints. citeturn5search11turn5search2 There is also published work extending PCGRL ideas into 3D domains (e.g., Minecraft-like tasks), which is conceptually close to your “world as substrate” approach. citeturn5search8 *Implementation sequence:* (1) define the state/action space for layout edits; (2) define multi-objective reward; (3) run offline training using logged traces first; (4) graduate to online experimentation in a sandbox. *User-facing result:* an operator can ask for “optimize this room for throughput” and receive a proposed rearrangement as a diff. *Risk/complexity:* medium–high; reward design and evaluation stability are non-trivial.

**Deliverable: extend the world model representation from flat vectors to relational structure.** *What exists now:* the environment is naturally a graph (objects + relations). *What needs to be built:* graph-based encoders (scene graph → latent) so Dreamer’s RSSM observes relational/spatial structure more faithfully. DreamerV2/V3 are built around recurrent state-space world models and imagination-based actor-critic learning, but adapting encoders/decoders for structured graphs is the real work. citeturn6search15turn7search0 *Implementation sequence:* (1) define a canonical relation set (near, inside, linked-to, provides-service); (2) build a graph encoder; (3) train predictive tasks first (next-state prediction) before policy learning. *User-facing result:* the system can reason about “adjacency” and “routes” rather than only coordinates. *Risk/complexity:* high; this becomes research-grade representation learning.

**Research Problems**

**Deliverable: evolved world-building (generator/solver loops) that produces playable layouts automatically.** *What exists now:* you have an execution engine and an evaluator (rewards). *What needs to be built:* a generator–solver training loop where the generator proposes worlds and the solver (agents) must succeed, rewarding “challenging but solvable” designs. This style is studied in adversarial RL for procedural content generation, where environment generation is rewarded based on solver performance. citeturn5search13 *User-facing result:* new worlds “grow” from training rather than being hand-authored. *Risk/complexity:* very high; this requires careful safety/quality constraints, or it will generate degenerate content.

## Multimodal Integration in Physical Space

**Quick Wins**

**Deliverable: VLM-based scene inspection snapshots.** *What exists now:* you can render the scene; you have a VLM modality; you have semantic memory (FelixBag). *What needs to be built:* “capture view → VLM describe → store caption/findings as an artifact + memory item,” enabling searchable, multimodal scene history. *Implementation sequence:* (1) add “snapshot-to-image” pipeline; (2) call VLM; (3) store results with snapshot ID and object references. *User-facing result:* the operator can ask “what changed since yesterday’s snapshot?” and get a grounded answer with references. *Risk/complexity:* low–medium; quality depends on consistent camera framing and stable object IDs.

**Medium Builds**

**Deliverable: SayCan-style affordance gating for tool selection in the world.** *What exists now:* workflows can chain tool calls; tools have observable success/failure and can be scored; you track provenance. *What needs to be built:* an “affordance scorer” that evaluates which tools are feasible in the current world state, and then combines that with LLM-proposed steps. **Deliverable: Voyager-style “skill library” stored as executable workflows + retrieval.** *What exists now:* you already have DAG workflows, semantic memory, and proven knowledge transfer; this maps naturally to a skill library. *What needs to be built:* a standardized “skill packaging” format: each skill is (a) a workflow template, (b) required facilities, (c) typed surfaces, (d) success detectors. **Research Problems**

**Deliverable: embeddings-as-spatial-reasoning bridge (semantic goals → object relations → coordinates).** *What exists now:* you have embeddings and structured object metadata. *What needs to be built:* robust grounding from “find the infirmary” → “which objects match infirmary semantics” → “which coordinate/zone should we navigate to” → “what interactions are available.” *Implementation sequence:* build a scene graph text layer + retrieval + disambiguation dialogs. *User-facing result:* natural-language spatial querying that is reliable. *Risk/complexity:* high; disambiguation and grounding errors can cascade without careful UI/guardrails.

- **Steward/Ledger Facility**: manages roster, food, wages, equipment; outputs “ledger state.”  
- **Promoter Facility**: schedules bouts, manages reputation/economy; outputs “event plan.”  
*Implementation sequence:* (1) author facility blueprints; (2) bind provider roles (LLM + embedder; optionally VLM for inspection); (3) create surfaces (operator dashboard + end-user view); (4) bind these to environment objects (workstations + panels). *User-facing result:* rooms become functional: clicking the armory opens an armory surface; running “equip trainee” produces real artifacts. *Risk/complexity:* medium; the design challenge is keeping each facility minimal and composable.

**Deliverable: progression from static → interactive → living world with clear gates.** *What exists now:* you already have snapshots, workflows, provenance, and a 3D renderer. *What needs to be built:* define three explicit gates for the ludus:  
1) **Static + labeled** (assets + zones + portals).  
2) **Interactive** (facilities work, artifacts produced, operator can run workflows).  
3) **Living** (agents locomote, queue at facilities, handle schedules).  
*Implementation sequence:* bake gates into bundle profiles so you can ship “Stage 1” without autonomy. *User-facing result:* demos are reliable and incremental rather than “everything at once.” *Risk/complexity:* medium; it’s mostly product discipline.

**Research Problems**

**Deliverable: believable autonomous gladiator loop (training → fatigue → injury → recovery → promotion) that is both fun and auditable.** *What exists now:* autonomy primitives can be built, and Dreamer can learn from usage. *What needs to be built:* robust agent motivations, scheduling, and failure recovery that stays explainable to operators. *Implementation sequence:* iterative simulation tuning and behavior-policy design; heavy testing. *User-facing result:* the world feels inhabited rather than scripted. *Risk/complexity:* high; “believable autonomy” is a long-horizon design problem.
