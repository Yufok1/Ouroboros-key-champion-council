## Recovered Thread Direction That Should Be Preserved

These ideas were not fully captured in source, but they clearly survived the continuity pass and should be treated as live design intent.

### 1. Trades are the intake engine, not the beneficiary class

The corrected economic intuition is:

- trades operationalize and generate the pool
- holding is the stronger incentivized posture
- trading is transient and happens anyway
- the system should empower holders

This is the central correction.

### 2. The meme coin is being treated as a bridge mechanic

The coin is not only an object to speculate on.

It is being framed as:

- a bridge between resource surfaces
- a unit that can carry diagnostic and economic meaning
- a shared public interface between market, social, creator, and system state

That "money pass" / bridge logic is still live and should remain preserved.

### 3. `Platinum DKP` is the correct recovered distribution metaphor

The recovered model is:

- a scarce prize/action/right is claimed or won
- the winner pays in
- that payment is split across the body or the qualified raid force
- holding and contribution benefit from the same intake

That means the system is not:

- pure burn theater
- pure jackpot gambling
- pure creator extraction

It is a raid-split logic over a live coin community.

### 4. Raw activity is not the scoring target

The recovered thread already corrected this.

Not:

- most activity wins
- highest churn wins
- spam produces payout

Instead:

- most value-positive contribution wins
- stabilization matters
- holder behavior matters
- specialists can emerge without turning the system into a farm

### 5. Recovery logic should exist, but not as refund logic

The preserved direction was:

- do not simply "pay back losers"
- do preserve proven participants who got wiped out
- use controlled re-entry / resurrection mechanics
- lock or vest recovery rather than making it instant liquidity

That is a meaningful continuity seam and should not be lost.

### 6. The single-token access gate is still live, but currently parked

The idea of:

- one token in wallet
- bag-holder access
- demo/build/fork gating

was not deleted.

But it is not the immediate center right now.

The stronger immediate seam is:

- operationalize the creator-pool / holder distribution logic first

and keep the single-token access system on hold.
